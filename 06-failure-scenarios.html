부록 A · 장애 시나리오

④ 시스템 아키텍처의 실행 단위가 하나씩 죽었을 때 무엇이 멈추고, 무엇이 살아남고, 어떻게 복구되는가. 설계가 "정상 경로"만 그리면 구현 단계에서 반드시 터지므로, 컴포넌트별로 영향 · 감지 · 복구 · 데이터 손실을 못박아 둔다.
심각도는 치명 핵심 루프 정지 · 부분 일부 기능 정지 · 경미 지연·열화만.

1. 실행 단위 프로그램이 죽거나 재시작될 때

컴포넌트심각도무너지는 것살아남는 것감지복구데이터 손실
미디어 서버
Go · 스테이트풀		 치명 진행 중 SRT 연결 끊김 · 라이브 시청 중단 · DVR 되감기 불가 · 이후 구간 영구 유실 이미 S3에 올라간 세그먼트 · 카드 · 편집·업로드(과거 구간) 플러그인 heartbeat 단절 + 인제스트 다운 알람 프로세스 재기동 후 플러그인 자동 재연결. 세그먼트를 즉시 S3에 적재해 피해를 재시작 구간으로 한정 있음 — 다운 구간 영상
채팅 수집기
Java · Lease		 부분 자동 하이라이트 감지 중단(입력이 끊김) · 채팅량 그래프 정지 송출·시청·DVR · 핫키 수동 카드 · 편집·렌더·업로드 전체 Lease 갱신 중단 · Redis 집계 유입 0 다른 인스턴스가 Lease를 획득해 재연결. 재연결 시 워밍업 구간을 다시 거쳐 베이스라인 재학습 다운 구간 채팅(아카이브·감지 모두)
하이라이트 감지기
Java		 부분 자동 카드 생성 중단 채팅 수집·아카이브 계속 · 핫키 수동 카드 · 나머지 전부 카드 발행률 0 알람 재기동 후 Redis 집계를 다시 읽어 워밍업 → 발행 재개. 카드 멱등키가 있어 재시작 중복 발행이 흡수됨 다운 구간 자동 카드(채팅 원본은 S3에 남아 사후 재처리 가능)
Core API
Java · Spring		 치명 로그인·화면·카드 조회·편집 저장·승인·작업 생성 전부 · SSE 끊김 송출 수신과 S3 적재(미디어 서버 독립) · 이미 큐에 들어간 작업의 워커 처리 헬스체크 실패 · 5xx 급증 다중 인스턴스라 한 대 죽어도 나머지가 처리. 전면 다운 시 재기동 후 SSE는 DB 스냅샷 먼저 보내고 이후 이벤트를 이어 붙임 없음 — 커밋된 트랜잭션은 PostgreSQL에 있음
워커 (4종)
자막·미리보기·렌더·업로드		 경미 해당 종류의 작업만 적체(예: 렌더 워커만 죽으면 렌더만 밀림) 다른 세 워커는 각자 큐라 영향 없음 · 실시간 경로 전체 큐 길이·처리 지연 알람 · DLQ 유입 메시지가 큐에 남아 있어 재기동 시 이어서 처리. 반복 실패 건은 전용 DLQ로 격리되어 정상 건을 막지 않음 없음 — jobId·renderVersion 멱등으로 재처리 안전
OBS 플러그인
C++ · 클라이언트		 부분 송출 중단 = 방송 자체 종료 · 핫키 마킹 불가 이미 수신된 구간의 시청·편집·업로드 전부 heartbeat 단절 정상 종료(stop)와 비정상 단절을 구분 — stop이 오면 즉시 VOD 확정, heartbeat만 끊기면 grace 후 단절 처리 없음

2. 저장소 · 큐 상태를 가진 것이 죽을 때

컴포넌트심각도무너지는 것살아남는 것감지복구데이터 손실
PostgreSQL
원본 · SoT		 치명 사실상 전체 — 인증·권한·카드·편집·잡 상태 모두 읽고 쓸 수 없음 송출 수신·S3 적재 · CDN이 캐시한 재생 커넥션 실패 · 페일오버 이벤트 Multi-AZ 자동 페일오버(1~2분). 그동안 Core는 쓰기 거부·읽기 실패를 정직하게 반환(거짓 성공 금지) 없음(커밋 기준)
Redis
휘발 · SoT 아님		 부분 채팅 집계 소실 → 자동 감지 일시 중단 · SSE 실시간 푸시 끊김 · 세션 무효화 시 재로그인 화면은 전부 재구성 가능 — 카드·권한·잡 상태는 PostgreSQL, 채팅 원본은 S3 연결 실패 · Pub/Sub 발행 실패 primary 죽으면 replica 자동 승격. 감지기는 워밍업부터 다시. 웹은 SSE 끊기면 REST 폴백으로 스냅샷 재조회 휘발 데이터만 — 원본 손실 없음
SQS + DLQ
작업 소유권		 부분 새 작업 발행·소비 불가 → 자막·미리보기·렌더·업로드 정지 실시간 경로 전체(송출·시청·채팅·감지·카드) · 조회·편집 저장 발행 실패 · 소비 정지 관리형 서비스 복구 후 자동 재개. Outbox에 남은 작업 발행 건은 재발행되므로 유실되지 않음 없음
S3
객체		 치명 세그먼트 적재 실패 → 시청·DVR·VOD·편집 원본 접근 불가 · 렌더 산출물 저장 불가 카드 생성·조회(메타는 PostgreSQL) · 권한·승인 PUT/GET 오류율 급증 재시도·백오프. 장기화 시 인제스트를 중단하는 편이 낫다 — 못 저장한 구간을 방송된 것처럼 보이게 두면 안 됨 적재 실패 구간
CDN
재생 딜리버리		 경미 재생 지연·버퍼링 · 최악의 경우 시청 불가 수집·감지·카드·편집·업로드 전부(재생만 영향) 에러율·캐시 히트율 급락 오리진 직결로 폴백(부하 증가 감수). 세그먼트는 불변이라 캐시 재구성이 빠름 없음

3. 외부 시스템 우리가 고칠 수 없는 것

대상심각도무너지는 것대응사용자에게 보여줄 것
치지직 / SOOP 부분 채팅 수집 불가 → 자동 감지 중단 공식 API 실패 시 비공식 WS로 폴백 · 재연결 백오프 "자동 감지 일시 중단 · 핫키로 계속 표시할 수 있습니다"
YouTube 부분 업로드 실패 · 응답 전 단절 시 성공 여부 불명 불명 상태는 RECONCILIATION_REQUIRED로 두고 채널을 조회해 실제 업로드 여부를 확인한 뒤 재시도. 확인 없이 재업로드하지 않는다(중복 게시 방지) "업로드 상태 확인 중" → 확정 후 성공/실패
Google OIDC 치명 신규 로그인 불가 기존 세션은 유지되므로 이미 접속한 사용자는 계속 작업 가능 "로그인 일시 불가 · 진행 중 작업은 계속됩니다"
결제 PG P3 경미 신규 구독·갱신 결제 실패 재시도 · 유예 기간 부여 후 등급 강등. 결제 실패로 진행 중인 방송·업로드를 끊지 않는다 결제 실패 알림 + 유예 기한

4. 정합성이 깨지는 순간 "둘 다 성공"이 보장되지 않는 지점

지점순진하게 짜면설계된 방어
카드 저장 ↔ 실시간 알림 DB에 카드를 넣고 따로 이벤트를 쏜다 → 저장은 됐는데 알림이 유실되거나, 알림만 나가고 저장이 롤백됨 Transactional Outbox — 카드와 Outbox 레코드를 한 트랜잭션에 함께 커밋하고, 발행은 그 뒤에 Outbox를 읽어서 한다. 발행 실패는 재발행으로 해결된다
감지기 재시작 · 메시지 재전달 같은 피크 구간에 카드가 여러 장 생긴다 카드 멱등키 broadcastId + detectorVersion + peakWindowStart 유니크 제약
작업 재시도 같은 클립이 두 번 렌더되고 두 번 업로드된다 jobId 멱등 · renderVersion으로 산출물 식별 · 업로드는 채널 조회 후 재시도
편집자의 업로드 편집 권한만 있는 편집자가 만든 작업이 그대로 유튜브로 나간다 업로드 권한 Gate 2단 — 작업 생성 시 1차, 실제 업로드 직전 2차. 권한이 중간에 회수돼도 2차에서 막힌다
방송 종료 판정 SRT가 끊기면 방송이 끝난 줄 알고 VOD를 확정한다 → 네트워크 순단이 방송 종료가 됨 플러그인의 명시적 stop은 즉시 종료 · heartbeat만 끊기면 grace 시간 후 단절로 판정하고 재연결을 기다린다
Core 다중 인스턴스 SSE 이벤트를 인스턴스 하나만 받아, 다른 인스턴스에 붙은 클라이언트는 갱신을 못 받는다 Redis Pub/Sub 팬아웃 — 전 인스턴스가 이벤트를 받아 각자의 SSE 연결로 밀어 준다. 재연결 시엔 이벤트를 전부 재생하지 않고 DB 스냅샷 → 이후 이벤트 순서로 붙인다

5. 우선순위 구현 순서

  • 1. OBS 제어선 — start/stop/heartbeat. 이게 없으면 정상 종료와 단절을 영원히 구분 못 한다.
  • 2. 작업 큐 분리 — SQS 4큐 + 각 DLQ. 렌더가 CPU를 다 먹어도 업로드가 밀리지 않게.
  • 3. Outbox · 멱등성 — 카드/작업이 중복·유실되지 않는 최소 보장.
  • 4. 업로드 권한 Gate — 협업(P2)을 열기 전에 있어야 한다. 나중에 붙이면 그 사이가 구멍이다.
  • 5. 미디어 포맷·보관 — CMAF 일원화, 라이브 1시간 / VOD 60일 / 임시 산출물 1~7일.
