## 시퀀스 다이어그램

**①②③④의 모든 카드가 눌립니다.** 카드마다 `SEQ n` 배지가 붙어 있고, 누르면 오른쪽 서랍에 그 카드와 관련된 처리 흐름 목록이 열립니다. 항목을 고르면 전체 화면으로 시퀀스 다이어그램이 뜹니다.

| 다이어그램 | 카드 | 카드가 뜻하는 것 |
|---|:-:|---|
| ① 유스케이스 | 32 | UC-01 ~ UC-32 |
| ② IA | 32 | 표면 9(컬럼 머리) + 화면 23 |
| ③ 서비스 | 29 | 논리 Capability 21 + 접점 4 + 외부 4 |
| ④ 시스템 | 20 | 실행 단위 · 저장소 13 + 접점 3 + 외부 4 |

시퀀스는 **45종**이고 정상 흐름뿐 아니라 실패·재시도·복구 흐름을 함께 담았습니다.

| 묶음 | 흐름 |
|---|---|
| `SEQ-PUB` 3 | 공개 웹 · 문의 · 서비스 상태 고지 |
| `SEQ-AUTH` 2 | 구글 로그인 · 계정 삭제(구독 해지 선행) |
| `SEQ-LINK` 3 | 방송 채널 · 유튜브 채널 연동 · 토큰 만료와 재인증 |
| `SEQ-PLUG` 2 | 플러그인 설치·연결 테스트 · 핫키 수동 마킹 |
| `SEQ-BC` 3 | 송출 시작 · heartbeat 단절과 유예 · 라이브 시청 |
| `SEQ-CHAT` 5 | 채팅 수집 활성화 · 집계·보관 · WS 단절 · Lease 인계 · 종료 Flush |

주소로 바로 열 수 있습니다. 새로고침해도 같은 화면이 뜨고, 브라우저 뒤로/앞으로도 동작합니다.

키보드로도 됩니다 — 카드에서 `Enter`/`Space`로 열고, `Esc`를 누르면 시퀀스 → 서랍 → 원본 순서로 한 겹씩 닫힙니다. 포커스는 닫은 뒤 원래 카드로 돌아옵니다.

### 같은 흐름을 카드마다 만들지 않는 구조

```
sequences.js     엔진 — 참여자 사전 · SVG 렌더러 · 카드 역인덱스 (데이터를 모른다)
seq-catalog.js   시퀀스 내용이 있는 유일한 곳
seq-ui.js        카드 클릭 → 목록 → 상세. 네 문서 공용 (CSS·DOM을 스스로 만든다)
```

카드 → 시퀀스 매핑은 손으로 적지 않습니다. 시퀀스가 스스로 `relatedUcIds` · `relatedIaNodeIds` · `relatedServiceIds` · `relatedComponentIds`로 "나는 어느 카드에 걸리는가"를 밝히고, 엔진이 역인덱스를 만듭니다. 그래서 `SEQ-CHAT-03` 하나가 ① `UC-13`, ② `라이브 대시보드`, ③ `채팅 수집`, ④ `채팅 수집기`·`Redis`·`Core API`·`치지직/SOOP`에 동시에 나타나되 내용은 한 벌만 존재합니다.

검사는 `node tools/check-catalog.js`로 돌립니다 — 카드 누락, 없는 카드 참조, 프래그먼트 불균형, 생명선에 없는 참여자 참조, 확정되지 않은 큐 이름, 초기 구성에 새어든 Kafka를 잡습니다.

## ④의 기술 기준

④는 팀 ADR과 확정 스택을 따릅니다. **PostgreSQL(+JSONB)** · **Java 21 Spring Boot/WebFlux** · **Go+FFmpeg** 미디어 · **Redis** 실시간 집계 · **SQS+DLQ** 작업 큐 · **CMAF/S3**.

**별도 이벤트 버스(Kafka 등)는 쓰지 않습니다.** 이벤트 팬아웃은 **Redis Pub/Sub**, 비동기 작업은 **SQS+DLQ**가 소유합니다. 이 두 개로 충분한 구성이라 메시지 브로커를 따로 두지 않습니다.

**성능 목표는 서버가 통제할 수 있는 구간으로만 잡습니다** — 채팅 수집 시각 → 카드 표시 p95 2초, 피크 감지 → 카드 표시 p95 1초. "장면 발생 → 카드 3초"는 플랫폼 방송 지연과 시청자 반응 시차가 끼어 서버가 보장할 수 없어 지표에서 뺐습니다(①③의 문구도 함께 정정).

## 개발 단계

각 문서 상단의 `P1` `P2` `P3` 칩을 누르면 **그 단계에 만드는 것만** 남습니다. 여러 개를 함께 켤 수 있고, 셋 다 켜면 전체와 같습니다.

| | 단계 | 기준 | UC |
|---|---|---|:-:|
| **P1** | 핵심 루프 | 스트리머 혼자 방송 중에 클립을 올린다 | 12 |
| **P2** | 협업 · 자산 | 편집자를 붙이고 방송이 끝나도 쓴다 | 13 |
| **P3** | 상용화 | 돈을 받고 모르는 사람이 가입해 쓴다 | 7 |

단계는 **①의 유스케이스에만 매기고 ②·③·④는 UC 번호로 물려받습니다.** 한 곳만 고치면 문서들이 같이 움직입니다. 화면이나 블록이 여러 UC에 걸치면 **가장 이른 단계**를 씁니다 — 그때 만들기 시작하고 기능 일부는 나중에 붙는다는 뜻입니다.

예외가 하나 있습니다. `AI 자막`은 UC-22(클립 편집, P1)에 속하지만 자막 기능만 P2에 붙이므로, ①의 UC-22 하위 배지 3개 · ②의 자막 패널 · ③의 AI 자막 블록을 모두 P2로 직접 지정했습니다.

**④는 단계 분해가 다르게 나옵니다.** 자동 감지 핵심 루프가 7개 프로그램 전부를 건드리므로, **시스템 골격은 P1에서 사실상 완성**됩니다. 그래서 ④에서 `P2`를 누르면 새로 켜지는 게 없습니다(협업·승인·AI 자막 등은 기존 프로그램 **내부** 기능 확장이라 새 실행 단위가 아님). 시스템 층에서 새로 붙는 요소는 `P3`의 결제(PG) 연동뿐입니다 — "MVP라고 인프라를 반만 세울 수는 없다"는 사실을 그대로 보여줍니다.

## 문서 사이의 규칙

- ②·③·④는 ①의 유스케이스 번호(`UC-01`~`UC-32`)로 ①을 참조합니다. 번호가 바뀌면 함께 고쳐야 합니다.
- ③의 서비스 블록은 ①의 유스케이스 32개를 **빠짐없이** 나눠 가집니다. 한 유스케이스가 여러 블록에 걸치는 경우(예: `UC-22` 클립 편집 = 편집 + 렌더 + AI 자막)는 서비스 경계를 가로지르는 것이라 의도한 것입니다.
- ④는 ③의 서비스 블록 21개를 **실제 실행되는 프로그램 7개로 N:M 매핑**합니다(예: `송출 수신` = 미디어 서버 + Core API). ③이 "무엇을 하는가"(논리 기능)라면 ④는 "무엇이 돌면서 어떤 프로토콜로 주고받는가"(실행 단위·전송)입니다. 감지는 ③·④ 모두 **채팅 기반 감지 서버**의 몫이고 플러그인은 핫키 신호만 보냅니다.
- 서비스 설명에 없는 기능·컴포넌트는 임의로 넣지 않습니다. 판단이 필요한 항목은 문서 안에 `TBD`로 남깁니다.
- 용어는 ADR-012의 공식 명칭을 따릅니다 — 업로드 권한은 **기본 / 직접 업로드 / 원클릭 자동 업로드** 3단계이고, "자동 업로드"라는 별도 개념은 없습니다(원클릭 자동 업로드 하나).
