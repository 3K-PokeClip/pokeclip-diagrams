부록 C · 상태 머신

방송 · 클립 · 업로드의 상태와 전이를 코드가 그대로 따라야 하는 규격으로 못박는다. 상태 전이는 전부 Core API가 PostgreSQL 트랜잭션으로 기록하고, 화면에 알릴 전이는 같은 트랜잭션의 Outbox로 발행한다(④). 빗금 = 시작 상태 · 이중 테두리 = 종료 상태(더 이상 전이 없음). 화살표 라벨 = 트리거.

C-1. 방송 (Broadcast) stop과 heartbeat 만료는 다른 길 — 순단은 방송 종료가 아니다

전이트리거부작용비고
IDLE → LIVESRT 연결 + startBroadcastStarted → 채팅 수집 Lease 시작 · 알림ingestEpoch 채번
LIVE → RECONNECTINGheartbeat 만료화면에 "재연결 중" 표시 · 수집은 계속유예(grace) 시간 내 복구 대기
RECONNECTING → LIVE플러그인 재연결세그먼트 이어붙임같은 broadcastId 유지
RECONNECTING → ENDING유예 실패BroadcastStopped · vod-finalize-job 발행 · 채팅 수집 중지비정상 단절로 기록
LIVE → ENDING플러그인 stop정상 종료 — 즉시
ENDING → FINALIZING미디어 서버가 job 소비S3 세그먼트 + PG 방송 상태만으로 매니페스트 생성멱등 broadcastId+ingestEpoch · 어느 인스턴스든 동일
FINALIZING → FINALIZING실패SQS 재전달로 재시도멱등이라 중복 안전 · 반복 실패는 DLQ+알람
FINALIZING → VOD_READY매니페스트 확정VOD 공개 · 만료 카드 챕터 재활성화(UC-15)60일 수명 시작
VOD_READY → EXPIRED60일 경과만료 임박 알림(사전) · 재출력 불가 전환클립·편집본은 유지(ADR-004)

C-2. 클립 (ClipProject → 렌더) 실패는 격리하고, 재시도는 멱등으로

전이트리거부작용비고
DRAFT → SAVED편집본 저장(UC-23)편집 설정 저장 — 자동 임시 저장과 별개ClipProject 소유: 클립 편집
SAVED → DRAFT재편집 · 재출력편집기 재진입재출력은 VOD 60일 내
DRAFT → RENDER_QUEUED렌더 큐 등록RenderRequested · render-jobs 발행jobId+renderVersion 채번
RENDER_QUEUED → RENDERING렌더 워커 소비작업 상태 보고(REST) 시작visibility timeout 30분+heartbeat 연장
RENDERING → RENDERED렌더 완료RenderCompleted · 산출물 S3 · 클립 자산 등록영구 보관(ClipAsset)
RENDERING → RENDER_FAILED렌더 실패실패 사유 기록 · 보관함에 재시도 노출최대 수신 초과 시 DLQ 격리
RENDER_FAILED → RENDER_QUEUED사용자 재시도동일 renderVersion 재발행멱등 — 중복 산출물 없음
RENDERED → DELETED사용자 삭제(UC-28)메타 삭제 상태 · S3 객체 수명 처리 · 삭제 이력 audit_log유튜브 업로드본 영향 없음

C-3. 업로드 (UploadAttempt) 불명확 성공은 RECONCILING — 확인 없이 재업로드하지 않는다

전이트리거부작용비고
REQUESTED → QUEUEDGate 1차 통과(직접 업로드 이상 권한)upload-jobs 발행UC-25·26 공통 진입
REQUESTED → PENDING_APPROVALGate 1차 — 권한 부족UploadRequested · 승인 대기함 · 스트리머 알림차단 대신 대기(ADR-012)
PENDING_APPROVAL → QUEUED스트리머 승인(UC-30)UploadApproved · 승인 이력 audit_log
PENDING_APPROVAL → REJECTED스트리머 반려반려 사유 · 편집자 알림 · audit_log종료 상태
PENDING_APPROVAL → CANCELED대기 중 권한 회수 · 편집자 취소요청 무효 · audit_log종료 상태
QUEUED → UPLOADING업로드 워커 소비 + Gate 2차 통과YouTube API 호출 시작2차 검사는 실행 직전
QUEUED·UPLOADING → BLOCKEDGate 2차 실패(권한 중도 회수)차단 기록 · audit_log · 스트리머·편집자 알림업로드는 나가지 않음
UPLOADING → UPLOADED성공 응답UploadCompleted · 비공개 + CC · 유튜브 링크 저장종료 상태
UPLOADING → FAILED명확한 실패(4xx · 토큰 만료 등)실패 사유 · 재인증 유도(UC-27)
UPLOADING → RECONCILING응답 전 단절 · 타임아웃 — 성공 여부 불명확재업로드 보류 · 채널 조회 예약무조건 재시도 금지(중복 게시 방지)
RECONCILING → UPLOADED채널 조회: 영상 존재성공으로 확정 · 링크 저장멱등 확정
RECONCILING → FAILED채널 조회: 없음 확인실패로 확정이제야 재시도 가능
FAILED → QUEUED재인증 후 수동 재시도(UC-27)동일 jobId 재발행Gate 2차 다시 통과해야 함
FAILED → CANCELED사용자 취소audit_log종료 상태
공통 규칙 — ① 전이의 원본은 PostgreSQL(상태 컬럼 + 전이 기록), 화면 갱신은 Outbox → Redis Pub/Sub → SSE. ② audit_log 대상 전이: 승인·반려·차단·취소·삭제·권한 변경(표에 표시). ③ 여기 없는 전이는 불법 — 코드에서 상태 검증 없이 상태를 덮어쓰지 않는다. ④ 세 머신의 연결: 방송 ENDING → 클립 재출력 시한(60일)의 기점 · 클립 RENDERED → 업로드 REQUESTED의 전제.
